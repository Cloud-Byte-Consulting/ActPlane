\section{Evaluation}
\label{sec:evaluation}

We evaluate \sys{} along three axes. RQ1 measures end-to-end policy
compliance on directive-derived rules from the empirical corpus;
RQ2 quantifies per-event and end-to-end overhead; RQ3 tests
repository-grounded coding tasks on an OctoBench subset with the official
judge.

\begin{description}
\item[RQ1 (Policy compliance)] Compared with prompt-only, tool-layer,
  and feedback-ablation baselines, does \sys{} improve policy
  compliance for directive-derived rules across direct and bypass
  execution paths?
\item[RQ2 (Overhead)] What is the per-event and end-to-end overhead of
  \sys{}'s label propagation and rule checking?
\item[RQ3 (Repository-grounded feedback)] On a scaffold-aware
  repository-grounded coding benchmark, does \sys{}'s OS-level
  enforcement and semantic feedback improve official policy compliance?
\end{description}

\subsection{Experimental Setup}
\label{sec:eval:setup}

\paragraph{Hardware and kernel.}
All experiments run on a single machine with an Intel Core Ultra~9~285K
CPU (24~hardware cores), 125\,GiB RAM, Linux~6.15.11, ext4 filesystem,
and libbpf~1.x with \texttt{bpf\_loop} support. The live enforcement
runs require BPF-LSM active (\texttt{bpf} in
\texttt{/sys/kernel/security/lsm}); the RQ2 overhead artifacts record
the exact LSM state used for each performance run.

\paragraph{Agent environment.}
% TODO: Fill in agent versions.
The active agent harnesses are Claude Code (via the \sys{} feedback
hook) and OpenAI Codex CLI (via tool-error feedback). Exact versions are
recorded in per-run metadata.

\paragraph{Baseline systems.}
All baselines use the \emph{same} DSL rules; only the enforcement layer
differs. Tool-layer baselines are Python scripts that read the same
\texttt{rule.yaml} and implement DSL semantics at the tool-call level.
Kernel baselines are \sys{} with features selectively disabled. We compare
seven configurations: prompt-only, TL-1 (single-event tool guard), TL-N
(multi-event tool guard), app-level IFC, per-event eBPF, kernel IFC
without feedback, and full \sys{}. The tool-layer and app-level baselines
represent AgentSpec/Progent-style
guards~\cite{wang2025agentspec,shi2025progent} and FIDES/CaMeL-style
tool-call label tracking~\cite{costa2025fides,debenedetti2025camel}; the
kernel baselines isolate the contributions of cross-event state and
semantic feedback.

\subsection{Rule Set Construction}
\label{sec:eval:ruleset}

The empirical study identifies 607~OS-enforceable directives:
392~per-event and 215~cross-event. RQ1 uses this subset because these
directives can be represented at process, file, or network hooks.
Semantic-only and content directives are excluded: the former have no
system-observable counterpart; the latter require file-content inspection
rather than OS-level IFC. The evaluation pipeline enforces strict
separation of concerns across four actors.

\paragraph{Pipeline overview.}
The pipeline has five stages with strict actor separation.
A translation LLM reads the sampled directive text, the project's
\texttt{CLAUDE.md}, the cloned repository, and the DSL reference to
produce a rule file; it cannot see traces or ground truth.
A separate trace generator reads the directive and project
structure to produce violation and compliant trace files, each with a
ground-truth header; it cannot see the rule file, preventing circular
bias.
A deterministic script then wraps the triggering command from each
direct trace in \texttt{bash~-c}, subprocess, and compiled-binary
variants, with no LLM involvement.
The eval harness replays each trace under each system, collects
the system response, and issues one API call to a tested LLM, a smaller
model standing in for the agent, for the next-action decision.
Finally, a scorer compares the tested LLM's decision against
ground truth and classifies the outcome as TP, TN, FP, or FN.
Because the translation LLM and the trace generator are different actors,
translation blind spots do not hide in traces; because the tested LLM
never sees ground truth, it acts like a real agent.
All systems run on the \emph{same} agent-generated rules, so translation
errors are shared noise and differences between systems reflect
architectural capability. Figure~\ref{fig:policy-example} shows
representative per-event and cross-event rules.

\subsection{RQ1: End-to-End Policy Compliance}
\label{sec:eval:rq1}

\paragraph{Method.}
RQ1 uses a trace-conditioned benchmark derived from the OS-enforceable
directive corpus. The current paper-facing sample contains 38 statements.
For each statement we construct six manifest-listed traces: three benign
traces (\texttt{canonical\_compliant},
\texttt{allowed\_effect\_compliant}, and
\texttt{lookalike\_compliant}) and three violation traces
(\texttt{visible\_violation}, \texttt{script\_visible\_violation}, and
\texttt{opaque\_fixture\_violation}). This yields 228 traces and 912
system--trace cells across four systems: prompt-filter, tool-regex,
\sys{} without structured feedback (\textsc{ActPlane-opaque}), and full
\sys{}.

Each trace is a JSONL transcript with a ground-truth header followed by a
user prompt and concrete tool calls that establish the repo state. The
tested agent then continues for a bounded number of real Agent SDK tool
steps. All systems use the same trace, model, Docker copy-on-write repo
execution, and trajectory judge. The prompt-filter baseline applies a
stateless LLM classifier to proposed tool actions using the natural-language
directive; tool-regex applies the per-case raw tool-input regex policy; and
\sys{} runs the generated OS policy below the tool layer through the
eBPF/ActPlane runner. Ground truth and policy artifacts are not shown to the
tested agent.

A separate LLM trajectory judge assigns TP, TN, FP, or FN from the raw
runner result. TP means a valid violation trace received guardrail-visible
intervention or recovery; TN means a valid benign trace completed without
wrongful guardrail intervention; FP means a benign trace was wrongly blocked,
reported, failed, or steered away; FN means a violation trace was missed or
left unrecovered. We report directive compliance rate
\(\mathrm{DCR}=(\mathrm{TP}+\mathrm{TN})/(\mathrm{TP}+\mathrm{TN}
+\mathrm{FP}+\mathrm{FN})\). For this snapshot, each
(\textit{system}, \textit{statement}, \textit{trace}) cell uses the latest
judged result matching the current trace artifact; all 912 cells are present
and none are stale relative to the trace files.

\sys{} improves overall compliance by 18.4 percentage points over
tool-regex and 20.1 points over prompt-filter. The improvement comes from
two complementary effects. First, \sys{} observes runtime side effects below
the tool layer, so it can detect subprocess and fixture-mediated violations
that are not available to prompt or raw tool-input filters. Second, it
reduces over-blocking on benign near-boundary traces: full \sys{} has 16
FPs, compared with 33 for prompt-filter and 29 for tool-regex.

The feedback ablation isolates the value of semantic recovery information.
\textsc{ActPlane-opaque} has very few false positives, but it misses many
violations (72 FNs) because the agent receives no structured corrective
payload after enforcement. Full \sys{} cuts those FNs roughly in half while
preserving substantially lower FP counts than the prompt and regex
baselines. Thus, the result is not simply that kernel enforcement can stop
more effects; model-facing feedback is part of the end-to-end guardrail.

\subsection{RQ2: Overhead}
\label{sec:eval:rq2}

\subsubsection{Macrobenchmarks (End-to-End Workloads)}

We measure end-to-end overhead on two deterministic workloads under
no-hit \sys{} configurations. The agent trace suite replays
20~corpus-derived traces (68~tool actions, 20~Bash subprocesses). The
Linux kernel build compiles \texttt{defconfig}~+~\texttt{vmlinux} with
\texttt{make~-j24} in a clean output directory. Each workload runs three
trials per configuration. Fixed workloads eliminate LLM inference variance
and isolate \sys{}'s runtime cost.

\begin{figure}[tb]
\centering
\includegraphics[width=\columnwidth]{figures/rq2_macro_overhead.png}
\caption{End-to-end overhead on deterministic workloads, normalized to
native execution. At 32 active no-hit rules, \sys{} adds 1.9\% on
agent-trace replay and 6.5\% on a Linux build; at 100 rules, overhead
remains below 8.4\%.}
\Description{Grouped bar chart showing AP-32 and AP-100 overhead relative
to native execution for agent trace replay and Linux build workloads.}
\label{fig:rq2-macro}
\end{figure}

\subsubsection{Microbenchmarks (Per-Syscall Latency)}

We measure \sys{}'s per-event latency for five syscall types
(\texttt{fork}, \texttt{exec}, \texttt{open}, \texttt{write},
\texttt{connect}) under five no-hit configurations: native, and \sys{} with 1, 10, 32,
and 100 active rules (AP-$N$ denotes \sys{} with $N$ rules). Each configuration $\times$ syscall type runs
10K--100K iterations pinned to a single CPU core. For each trial we
compute p50 and p99 latencies; Table~\ref{tab:micro-latency} reports
the median across seven trials.

\paragraph{Measured operations.}
\texttt{fork}: \texttt{fork()} + parent \texttt{waitpid()}.
\texttt{exec}: \texttt{fork()} + child \texttt{execve("/bin/true")} +
parent \texttt{waitpid()}.
\texttt{open}: \texttt{open(path, O\_RDONLY|O\_CLOEXEC)} with
\texttt{close()} outside the timed region.
\texttt{write}: \texttt{write(fd, 4096)} to an already-open temp file.
\texttt{connect}: UDP \texttt{connect(127.0.0.1:9)} on an
already-created socket.

\paragraph{Results.}
The largest per-call cost is on \texttt{open}: AP-100 adds
12.8\,\textmu s at p50 (23$\times$ native), because each open triggers a
path hash (FNV-1a), a file-label map lookup, and a linear scan of up to
100~rules. \texttt{connect} adds 2.6\,\textmu s and \texttt{write}
0.57\,\textmu s at p50; both involve shorter matching paths.
\texttt{fork} and \texttt{exec} include scheduler and process-management
overhead, so their tail latencies are noisier. Despite the high per-call
multiplier on \texttt{open}, the macrobenchmark impact remains low
(1.9\%--8.4\%) because the absolute added cost (12.8\,\textmu s) is
small relative to the computation, I/O, and scheduling time between
successive opens. The per-call overhead therefore does not accumulate
into a significant fraction of end-to-end wall-clock time.

label propagation and rule scanning without violation reporting or

\subsection{RQ3: Repository-Grounded Policy Compliance (OctoBench)}
\label{sec:eval:rq3}

\paragraph{Goal.}
Unlike RQ1, which isolates single decision points, RQ3 evaluates
\sys{} on complete coding-agent tasks in real repository environments,
scored by the official OctoBench whole-case checklist judge.

\paragraph{Benchmark and subset.}
OctoBench~\cite{octobench} is a scaffold-aware repository-grounded
benchmark with 217~Dockerized tasks spanning system prompts, tool
schemas, project files (\texttt{CLAUDE.md}, \texttt{AGENTS.md}), and
user queries. We use the official evaluator unchanged. Because \sys{}
cannot observe purely semantic checks (e.g., tone), we select a 20-task
subset with system-observable policy points and case-specific policies.
Each task runs under three conditions: baseline (no enforcement),
tool-regex (case-specific tool-call hooks), and \sys{} (OS-level hooks
plus semantic feedback).

\paragraph{Metrics.}
The primary metric is official OctoBench reward (fraction of checklist
policies judged satisfied). We additionally report three diagnostic
submetrics from the same checklist results: \emph{user-query reward}
(user-task checks only), \emph{implementation/test reward}
(implementation, testing, configuration, and modification checks), and
\emph{compliance reward} (compliance-typed checks). All submetrics use the
official LLM-based checklist judge.

\begin{figure}[tb]
\centering
\includegraphics[width=\columnwidth]{figures/octobench_rq3.png}
\caption{OctoBench 20-task selected subset. \sys{} improves official
policy compliance over both baseline and tool-regex.}
\Description{Grouped bar chart comparing baseline, tool-regex, and
\sys{} on official reward, user-query reward, implementation/test
reward, and compliance reward.}
\label{fig:octobench-rq3}
\end{figure}

\paragraph{Results.}
On the 20-task subset, official reward is 0.788 (baseline), 0.818
(tool-regex), and 0.888 (\sys{}): a 10.0-point gain over baseline and
7.0~points over tool-regex. The improvement extends beyond generic
compliance: user-query reward rises by 28.2~points and
implementation/test reward by 25.0~points over baseline. These results
support the claim that OS-level enforcement with semantic feedback
improves policy compliance in repository-grounded coding agents. We note
that OctoBench uses an LLM checklist judge rather than deterministic test
scripts, so we do not claim deterministic task completion.

\section{Related Work}
\label{sec:related}

\textbf{In-kernel provenance and information flow.}
Whole-system provenance systems such as PASS~\cite{muniswamy2006pass},
Hi-Fi~\cite{pohly2012hifi}, LPM~\cite{bates2015lpm}, and
CamFlow~\cite{pasquier2017camflow} record audit-grade provenance
graphs via LSM hooks but do not compile agent-oriented policy or return
semantic feedback. CamQuery~\cite{pasquier2018camquery} is the closest
prior system: it evaluates provenance queries inline in the kernel,
propagates labels across OS objects, and can deny matching operations.
\sys{} shares this label-propagation model but targets cooperative AI
agents rather than adversarial intrusions, compiles an agent-facing
source/sink DSL rather than general provenance queries, and returns
semantic feedback to the matching actor. Dynamic taint-tracking
systems~\cite{enck2010taintdroid,clause2007dytan,kemerlis2012libdft,yin2007panorama}
operate at byte or instruction granularity for vulnerability detection;
\sys{} operates at object granularity, trading precision for low-overhead
whole-session coverage.

\textbf{eBPF enforcement and agent guardrails.}
BPF-LSM~\cite{singh2019krsi,linuxbpflsm} lets eBPF programs attach to
security hooks and return allow/deny verdicts; \sys{} uses it as its
pre-operation enforcement backend. Contemporary eBPF tools such as Tetragon,
Tracee, and eBPF-PATROL~\cite{ciliumtetragon,aquasecuritytracee,ghimire2025ebpfpatrol} provide
per-event predicates or single-channel lineage flags; \sys{} propagates
multi-label information flow across channels and checks cross-event
conditions. At the agent layer, AgentSpec and
Progent~\cite{wang2025agentspec,shi2025progent} enforce deterministic
guards at the tool-call boundary; \sys{} complements them below the tool
API, where shell-outs and subprocesses remain visible. FIDES and
CaMeL~\cite{costa2025fides,debenedetti2025camel} apply typed IFC within
the agent loop but do not expose a layered authority stack that separates
platform, project, and agent policy; Flume~\cite{flume} provides
decentralized labels with explicit capabilities but requires
per-application integration rather than a compile-time authority
hierarchy. \sys{} applies information-flow reasoning at the OS boundary
with a layered authority model, where observation persists across context
windows and cannot be circumvented by subprocess indirection.

\textbf{Agent instruction files.}
Recent studies characterize \texttt{CLAUDE.md} and \texttt{AGENTS.md}
files along dimensions of content taxonomy, maintenance practices, and
task
efficiency~\cite{chatlatanagulchai2025manifests,chatlatanagulchai2025agentreadmes,
santos2025claudeconfig,lulla2026agentsmd}. These works classify at file or
section-heading granularity. Our empirical study
(\S\ref{sec:empirical}) classifies at statement level along four axes,
specifically asking which instructions require cross-event enforcement
and which need project or task context to instantiate.

\section{Conclusion}
\label{sec:conclusion}

An empirical study of 64~projects shows that agent instruction files are
behavioral policies whose enforcement demands OS-level observation,
cross-event state, and agent-provided context. \sys{} addresses this by
letting agents author labeled information-flow policies that an eBPF/BPF-LSM
engine enforces below the tool layer, returning semantic feedback when rules
match. A layered authority model prevents agents from weakening
higher-authority constraints. \sys{} does not cover semantic-only
directives (17\%) or content-inspection directives (38\%) that require a
linting layer, and the cooperative threat model excludes adversarial
evasion. Extending coverage to content-level checks and multi-agent
domains are natural next steps.
